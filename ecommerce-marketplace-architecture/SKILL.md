# 🛒 E-commerce & Marketplace Architecture Mastery

This skill defines the architectural dogmas and absolute best practices for building enterprise-grade **E-commerce** and multi-vendor **Marketplace** platforms.

## 🏗️ Core Architectural Dogmas

### 1. Domain-Driven Design (Bounded Contexts)
*   **Dogma:** A monolith eCommerce platform will eventually collapse under its own weight. You MUST separate the platform into logical Bounded Contexts.
*   **Rule:** At a minimum, logically define boundaries for: `Catalog` (Products/Pricing), `Inventory` (Stock), `Cart/Checkout` (Orders), and `Payment/Billing`.
*   **Rule:** Each bounded context MUST own its data. The `Order` service does not join tables with the `Catalog` service natively; it stores a snapshot of the product price and name at the exact moment of purchase.

### 2. Distributed Transactions: The Saga Pattern
*   **Dogma:** You cannot use strict two-phase commit (2PC) database transactions across distributed microservices (e.g., locking Inventory and Payment simultaneously).
*   **Rule:** Implement the **SAGA Pattern** (typically Orchestration-based for Checkout).
*   **Workflow:**
    1.  Checkout Service receives order -> Sets status to `PENDING`.
    2.  Sends command to Inventory Service -> "Reserve Stock".
    3.  If Stock Reserved OK -> Sends command to Payment Service -> "Charge Card".
    4.  If Payment Fails -> Sends compensating command to Inventory -> "Release Stock", sets Order `FAILED`.
    5.  If Payment OK -> Order `CONFIRMED`.

### 3. Payment Splitting (Marketplaces)
*   **Dogma:** In a multi-vendor marketplace, you MUST NEVER receive the full payment into the platform's primary bank account and manually transfer it to vendors. This creates massive tax/compliance liabilities.
*   **Rule:** Use Payment Gateway Split APIs (like Stripe Connect or Asaas Split). The platform acts as the technical orchestrator.
*   **Rule:** The gateway natively routes `X%` to the Vendor Account and `Y%` to the Platform Account (Take Rate / Fee) before the money even settles.

## ⚙️ Consistency and Scale

### 1. Idempotency Keys at Checkout
*   **Dogma:** Users will double-click the "Buy" button. Network requests will drop and retry.
*   **Rule:** Every Checkout/Payment mutation API MUST require an `X-Idempotency-Key` header (usually a UUID generated by the frontend). The backend MUST check if this key was already processed. If yes, return the previous successful response. Do NOT charge the card twice.

### 2. Inventory Eventual Consistency vs. Hard Locks
*   **Dogma:** Do not lock the Inventory database row when a user simply puts an item in their cart.
*   **Rule:** Implement soft reservations (e.g., holding stock in a fast Redis cache with a 15-minute TTL when the user enters the checkout funnel). Hard deduction only occurs on successful payment authorization.
